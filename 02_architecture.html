Agentic Media Pipeline — Architecture

A modular agent orchestration for B2B short-form ad production. Six specialist agents, one stateful graph, and the single hardest bet in the system: making a brand mascot look like the same character across thirty different generated frames.

§ 01 · Framing

CTO mode — what we're actually building.

Strip away the brand and the pitch. The system is a stateful workflow engine that takes a structured campaign brief and produces a set of short-form video assets, with three properties most "AI video" pipelines don't have:

  1. Identity persistence. The same mascot/hero/product appears recognisably across every frame in every variant of the campaign.
  2. Critique-gated output. No asset reaches the customer without passing an LLM-as-judge quality check against the brief and the brand rubric.
  3. Cost predictability. Every job has a hard upper bound on token + GPU spend, set by the customer's tier. The orchestrator enforces it; agents cannot exceed it.

Everything else — script generation, dub, edit, aspect-ratio mux — is well-trodden plumbing on top of vendor APIs. The defensibility lives in 1 and 2. Most of the engineering effort goes there.

Architectural bet We are not training models. We are not fine-tuning models. We are a workflow + state + judgment layer over commodity inference. If you (the engineer reading this) want to fine-tune a base diffusion model, you are in the wrong company.
§ 02 · Orchestration framework

Why LangGraph — and not CrewAI.

The choice between LangGraph and CrewAI is the most consequential framework decision. I'm picking LangGraph. Reasoning, then the counter-argument:

CapabilityLangGraphCrewAIWhy it matters here
Explicit state schema First-class Implicit / context-bag Character consistency requires a known, queryable state object. CrewAI's "shared context" is a hash bag. LangGraph forces you to declare what state is.
Conditional routing Native (graph edges) Manager-agent pattern Critique → regenerate is a conditional edge. LangGraph models it cleanly. CrewAI requires a Manager LLM to decide, which adds tokens and non-determinism.
Replay / checkpointing Built-in (SqliteSaver, Postgres) DIY When a job fails 70% through after burning ₹180 in API calls, we resume from checkpoint, not from zero.
Human-in-the-loop interrupt() primitive Possible, awkward Customer brief approval is HITL. We pause the graph, show the parsed brief, wait for approval, resume.
Vendor-agent abstraction Lower-level Higher-level CrewAI is faster to prototype if your agents are all chat-LLMs. We have non-LLM agents (Runway video gen, Sarvam dub). LangGraph's lower-level model fits non-LLM nodes better.
Learning curve Steeper Gentler Real cost. Adds 1–2 weeks to first prototype.
The honest counter CrewAI ships a working multi-agent prototype in a weekend. LangGraph takes a week. If we were running a hackathon, CrewAI wins. We're not. We're building a system where a single failed checkpoint costs a customer ₹500 in regenerated assets. The state-management primitives matter more than the prototyping speed.
§ 03 · The graph

The state machine — every job's spine.

One job = one graph traversal. The graph has six worker nodes (the agents), one router (Critique), one checkpoint after every node, and exactly two terminal states: delivered and killed.

Edges that matter

  • Brief Parser → HITL: We pause for customer to confirm parsed brief. Saves ~₹150 of wasted generation if the AI parsed the brief wrong.
  • Critique → regenerate: The dashed loop is where we eat our own cost. Bounded to max 2 regenerations per agent. After that, the job either ships with a quality flag or is killed and refunded.
  • Budget breaker: Independent watcher. If cumulative job cost exceeds 1.4× the tier cap, kill the job. Not a soft limit. A hard one.
§ 04 · State schema

The state schema.

This is the most important file in the codebase. Every agent reads from and writes to this object. If the schema is right, the system works. If it's wrong, no amount of clever orchestration saves us.

# state.py — single source of truth for any in-flight job

from typing import Optional, Literal, Annotated
from pydantic import BaseModel, Field
from datetime import datetime
import operator

class CharacterRef(BaseModel):
    """The most important object in the system.
    Everything character-consistency-related flows through this."""
    char_id: str                          # stable across whole job
    name: str                             # 'Mr Beans the mascot'
    source_image_urls: list[str]          # original customer-uploaded refs
    locked_image_url: str                 # canonical 'identity-locked' render
    ip_adapter_embedding: Optional[str]   # path/URI to the IP-Adapter conditioning
    seed: int                             # diffusion seed locked at identity time
    style_descriptor: str                 # short prose lock — e.g. '3D-rendered cartoon, soft lighting'
    constraints: list[str]                # 'always wears red scarf', 'no glasses'

class ShotSpec(BaseModel):
    shot_id: str
    char_refs: list[str]                  # which CharacterRef.char_ids appear
    duration_s: float
    camera: str                           # 'wide push-in', 'top-down product hero'
    setting: str                          # 'kitchen, morning light'
    action: str                           # one sentence of what happens
    dialogue: Optional[str]
    aspect: Literal['9:16', '1:1', '16:9']

class AgentResult(BaseModel):
    agent_id: str                         # '01-brief', '03-scene', etc.
    status: Literal['ok', 'retry', 'fail']
    output_uri: Optional[str]             # R2 path
    cost_inr: float                       # COGS, tracked per call
    elapsed_ms: int
    notes: str

class CritiqueVerdict(BaseModel):
    target_shot_id: str
    pass_: bool = Field(alias='pass')
    score: float                          # 0–1
    failures: list[str]                   # 'character drift in scene 3', 'audio out of sync'
    regenerate_agent: Optional[str]       # which agent to send back to

class JobState(BaseModel):
    # IDENTITY
    job_id: str
    customer_id: str
    tier: Literal['starter', 'studio', 'agency']
    created_at: datetime

    # INPUT
    brief_raw: str                        # what customer typed
    brief_parsed: Optional[dict]          # structured by 01-brief
    brief_approved: bool = False          # HITL gate

    # CHARACTERS — the spine of consistency
    characters: list[CharacterRef] = []   # locked once, read by all downstream agents

    # PLAN
    shot_list: list[ShotSpec] = []        # built by Brief Parser

    # OUTPUTS — accumulated, never overwritten
    agent_log: Annotated[list[AgentResult], operator.add] = []
    critique_log: Annotated[list[CritiqueVerdict], operator.add] = []
    asset_uris: dict[str, str] = {}       # shot_id → final muxed asset URI

    # BUDGET
    tier_cap_inr: float                   # set once at job creation
    cost_so_far_inr: float = 0.0          # updated by every node
    regen_count_by_agent: dict[str, int] = {}

    # TERMINAL
    final_status: Optional[Literal['delivered', 'killed', 'flagged']]
    kill_reason: Optional[str]
Why this schema, specifically Three deliberate choices. (1) characters is locked early and never mutated downstream — that's the consistency anchor. (2) agent_log uses LangGraph's operator.add reducer so each node appends rather than overwrites — full audit trail. (3) cost_so_far_inr is on the state object, not a side-effect — the budget breaker can read it inside the graph without external dependencies.
§ 05 · The hardest problem

Character consistency — where everyone fails.

This is where most AI ad-generation tools visibly break. The mascot's face shifts shape between scenes 1 and 4. The product's color drifts. The hero actress's hair changes from black to dark brown in the same campaign. Customers can spot this in three seconds. They will not pay for output that drifts.

Three layers of consistency control

Layer 1 — Identity Lock at job start

The Identity Lock agent runs once at the start of every job, never per-shot. Its job is to convert customer-uploaded references into a canonical conditioning artifact that all downstream rendering uses.

  • Customer uploads 3–8 reference images of mascot/product/hero.
  • We run them through Flux 1.1 Pro with IP-Adapter to extract a stable visual conditioning.
  • We generate one canonical "locked" reference image at a fixed seed and store it in R2.
  • The IP-Adapter embedding, the seed, and the canonical URL go into the CharacterRef object on state.

From this point forward, no agent generates a character from scratch. They all condition on the locked reference.

Layer 2 — Conditioning at every render

The Scene Renderer never calls Runway/Kling with just a prompt. It always calls with prompt + locked_image_url + style_descriptor. For Runway Gen-4, this means using the image-to-video endpoint with the locked image as the first frame conditioner. For Kling, the equivalent is the "subject reference" parameter.

async def render_shot(state: JobState, shot: ShotSpec) -> AgentResult:
    chars = [c for c in state.characters if c.char_id in shot.char_refs]
    primary = chars[0]  # main character anchor

    response = await runway_client.generate_video(
        prompt=build_prompt(shot, chars),
        reference_image=primary.locked_image_url,    # <-- the lock
        seed=primary.seed,                           # <-- deterministic
        style_lock=primary.style_descriptor,
        duration=shot.duration_s,
        aspect=shot.aspect,
    )

    cost = response.metadata.cost_inr
    return AgentResult(
        agent_id="03-scene",
        status="ok",
        output_uri=response.output_url,
        cost_inr=cost,
        elapsed_ms=response.elapsed_ms,
        notes=f"shot={shot.shot_id} char={primary.char_id}"
    )

Layer 3 — The Critique gate catches what Layers 1–2 miss

Even with perfect conditioning, diffusion models drift. The Critique agent's most important job is not "is the asset good" — it's "does the character in this asset still look like the locked reference?"

We implement this with a structured LLM-as-judge prompt that takes the locked reference image + the rendered shot + the brief, and produces a structured verdict:

CRITIQUE_RUBRIC = """
You are a quality reviewer for a character-consistent ad pipeline.

You will see:
1. The LOCKED reference image of the character.
2. The rendered shot.
3. The shot brief.

Score each dimension 0–1:
- identity_match: Does the character in the shot look like the same person/mascot as the locked reference? Pay attention to face structure, hair, signature features, color palette.
- brief_fidelity: Does the shot do what the brief said?
- platform_compliance: Does it meet platform spec (aspect, duration, no broken artifacts)?

Only return JSON. If identity_match < 0.75, set pass=false and regenerate_agent='03-scene'.
"""
The honest limit Even with all three layers, character consistency in current-gen video models is good, not perfect. A skilled customer will spot drift in 1 of every 8–10 shots. The Critique gate catches most of those before delivery. The ones it misses are why the brand voice promises "if Critique misses it and you reject it, we regenerate free." That promise is technically backed by the regen-count counter on state — we know exactly what we owe per job.

What we explicitly don't do (yet)

  • We don't train per-customer LoRAs. Too expensive at v1, locks us into one base model.
  • We don't use Stable Video Diffusion or open-source video models for production. Quality gap vs Runway/Kling is too wide for paying customers.
  • We don't allow more than 2 distinct characters in a single shot in v1. Multi-character consistency is a v2 problem and a different research domain.
§ 06 · ComfyUI

Where ComfyUI fits — and where it doesn't.

You mentioned ComfyUI as part of the stack. Let me be direct: ComfyUI is not the orchestrator. LangGraph is. ComfyUI is one of the things LangGraph calls.

ToolJobWhere it lives
LangGraphOrchestrates the whole job. Owns state, checkpoints, routing.Python service, Cloudflare Workers or Supabase Edge.
ComfyUIExecutes complex multi-node image-gen workflows that we self-host (e.g. Flux + IP-Adapter + ControlNet chains).Self-hosted on a single GPU box (₹40–80k/mo for an A100 instance, or rent per-hour from RunPod).
Runway / Kling APIsHosted video generation. We do not self-host these.Vendor APIs called from Scene Renderer agent.
Sarvam / ElevenLabs APIsVoice / dub / lip-sync.Vendor APIs.
FFmpeg + RemotionFinal compositing & aspect-ratio mux.Same Python service or a separate worker.

When to use ComfyUI vs hosted APIs

  • Use ComfyUI for the Identity Lock step. You need precise control over IP-Adapter weights, ControlNet conditioning, and seed. Hosted APIs hide too many knobs.
  • Use ComfyUI for any image-gen step where we need a multi-node graph (mask + inpaint + upscale). fal.ai's Flux endpoint can't do that.
  • Don't use ComfyUI for video generation. Self-hosting Runway-quality video is currently impossible without an enterprise contract and 8× H100s. Just call the API.
  • Don't use ComfyUI as the orchestrator. It has its own workflow graph, but it's not designed for stateful, multi-vendor, multi-modal pipelines. You'll fight it.
Practical pattern Wrap ComfyUI behind a thin FastAPI service. Each agent that needs ComfyUI just POSTs a workflow JSON to that service and polls for the result. From LangGraph's perspective, it looks identical to any other vendor API. This isolates ComfyUI's quirks from the orchestration layer.
§ 07 · Agent contracts

Agent contracts & the critique gate.

Every agent implements the same interface. This is non-negotiable — it's what makes them swappable.

from abc import ABC, abstractmethod

class Agent(ABC):
    """Every agent is a pure function: state in, AgentResult out.
    No agent mutates state directly — LangGraph reducers do that."""

    agent_id: str
    max_retries: int = 2
    cost_ceiling_inr: float = 100.0   # per-call hard cap

    @abstractmethod
    async def run(self, state: JobState) -> dict:
        """Returns a dict of state fields to merge.
        Must include 'agent_log': [AgentResult]."""
        ...

    async def __call__(self, state: JobState) -> dict:
        # Budget guard
        if state.cost_so_far_inr + self.cost_ceiling_inr > state.tier_cap_inr * 1.4:
            return {
                "final_status": "killed",
                "kill_reason": f"{self.agent_id} would exceed budget",
            }
        return await self.run(state)

The Critique agent in particular

It's the only agent that can route the graph backwards. It returns not just AgentResult, but a CritiqueVerdict that the conditional edge inspects to decide where to go next.

def route_after_critique(state: JobState) -> str:
    last = state.critique_log[-1]
    if last.pass_:
        return "deliver"

    # How many times have we already regenerated this agent?
    agent = last.regenerate_agent
    count = state.regen_count_by_agent.get(agent, 0)

    if count >= 2:
        return "deliver_with_flag"   # ship it but mark it for review
    return agent                       # send back to the failing agent

The "deliver with flag" path matters. We don't infinitely regenerate. If after 2 attempts we still can't pass quality, we ship the best attempt with a quality flag in the customer-facing dashboard, and we eat the regen cost. This is a deliberate trade — better to ship something with a flag than spin forever burning API calls.

Stack hygiene rules

  • No Redis at v1. Postgres does queueing, caching, state. Adding Redis is a Year-2 problem.
  • No Kubernetes at v1. Railway / Fly.io is enough until 50+ paying customers.
  • No microservices. One Python service, multiple worker types. Decompose only when something specific demands it.
  • No ML infrastructure (Modal, Replicate's training tier, etc.) at v1. We're not training. We are calling.

§ 10 · Observability

What to instrument first.

Most engineering teams over-instrument. We will under-instrument and add as needed. Day-one minimum:

  1. Per-job COGS — every agent writes its cost into the agent_log. Roll up to a dashboard per customer per week.
  2. Per-agent regeneration rate — if Scene Renderer is regenerating >30% of the time, the Identity Lock is broken upstream.
  3. Critique pass rate by language — if Telugu lip-sync is passing 95% but Tamil is at 70%, we have a vendor-quality problem in one language.
  4. Customer rejection rate — assets that pass Critique but get rejected by the customer. This is the gap between our judgment and theirs. Closing it is the product roadmap.
  5. p50 / p95 job latency — promised SLA is 18h. If p95 drifts above 24h, we have an upstream bottleneck.

Everything else is a distraction until we have customers using the system.

§ 12 · The ruthless cut

What I'd cut if forced to ship in 4 weeks.

This is the "what's truly load-bearing" test. If I had 4 weeks instead of 12:

  • Drop the dashboard. Customer emails brief, we email back assets via Drive link. No app at all.
  • Drop multi-language. English-only at v0.1. Add Hindi at week 6.
  • Drop multi-aspect. 9:16 only. Add 1:1 and 16:9 once one customer is paying.
  • Drop the critique gate. I — the founder — am the critique. Manually review every asset before delivery for the first 20 jobs. Critique-as-LLM comes when manual review hits its limits.
  • Drop budget breaker. First 20 jobs, I eat the cost variance personally and learn the COGS distribution.
  • Drop HITL automation. I phone the customer to confirm the brief. Manual.
  • Keep: LangGraph, state schema, Identity Lock, Scene Renderer, FFmpeg mux, Razorpay.
The lesson Everything cut above is a system replacement for the founder's manual labor. At <20 jobs, the founder is cheaper than the system. After 20 jobs, automation pays for itself. Sequence matters: build the manual version first to learn the customer; then automate the parts that hurt most.
